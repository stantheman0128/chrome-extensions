- colonist-stats-tracker (1.141.0): **collapse the five WS-sync call sites onto two helpers, preserving the deliberate dice/no-dice split (tri-AI review ④).** Five places hand-wrote the `syncFromWS(); syncStatsFromWS(); [syncDiceFromWS();]` sequence with the same OR-aggregation boilerplate (`let ch = …; if (…) ch = true;`). Two patterns existed and the difference is load-bearing: the per-frame handler, the PLAYING tick, and the manual resync pull all three (hands + stats + dice), while the ENDED tick and `buildGameRecord` deliberately pull only hands + stats — dice are out of scope once the game is over / when snapshotting the record (the omission is commented at both sites). Rather than flatten that, the refactor adds `syncHandStatsFromWS()` (hands + stats) and `syncAllFromWS()` (hands + stats + dice), routing each site to the matching one. Each sub-sync still runs unconditionally (no `||` short-circuit, so a change in one never skips the next) and the return still aggregates with OR, so every site's change/return behaviour is identical. Pure de-duplication; 445 suite + 22 oracle, and the real-capture replays that drive these sync paths are unchanged.
- colonist-stats-tracker (1.140.0): **gate the DOM discard tally on `wsOwnsStats`, matching `gained` (tri-AI review consistency fix).** The DOM log handler incremented `discards` + `discardCards` for every "discarded" line even when the WS board already owns that player's stats, while the sibling `gained` column two branches up is gated on `if (!wsOwnsStats(player))`. `syncStatsFromWS` overwrites both discard columns from the board's type-55 accrual, and `wsOwnsStats`'s own comment names "discard" as a WS-owned column the DOM must not also increment — so the ungated path was an oversight that could let `onGameWon` archive a doubled discard figure before the next WS tick corrected the live value (the exact failure the `gained` gate exists to prevent). The discard branch now wraps the two tally increments in the same guard; the hand removal (`takeResource`) stays ungated because it is owned separately by `wsOwnsHand` inside the resource helpers. Pure consistency fix, no new arithmetic; the DOM-only fallback (no WS board) is unchanged, so the existing discard parsing tests stay green. 445 suite + 22 oracle.
