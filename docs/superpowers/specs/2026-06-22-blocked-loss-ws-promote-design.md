# ⛔ 卡田:遊戲中精準化(D)— 設計

## 重大發現:Stan 的「每回合存當下地圖」其實已經建好大半(1.65)

D 原本想成「新做一整套地圖時間線快照」。但讀完現有程式後,**那套機制的等價物已經在了**,只是沒被顯示:

- `board.js` 的 `accrueBlocked` 在**每次擲骰當下**跑:用「當時的強盜位置 + 當時的建築」算出這一輪每個玩家被擋幾張(村莊 1 / 城市 2)。這就是「用當時地圖算」——天生解掉「蓋好前還是蓋好後」(那一刻是村莊就算 1,事後升城市不回頭改)。
- 1.65 的 `accrueWsBlocked` 把這個即時累計**持久化**進 `state.wsBlocked`(baseline + 單調增量),所以 **reload 後不會掉**(伺服器只重送最終地圖、無法回推歷史,但我們存的是「已經算好的累計損失」,不需要回推)。

換句話說:**不必存整張地圖快照,存「每一刻算好的損失累計」就達到同樣精準,而且這版已經寫好、會跨 reload。** 這比原本想的省很多。

## 那為什麼現在沒在用它?

因為**從沒拿真實對局驗證過它準不準**。目前:

- 面板顯示的 ⛔ = 「實收對比」估計(局末用 colonist Victory 畫面精確值覆蓋)。
- `state.wsBlocked`(幾何即時累計)只當**背景備份**,在 `__cstAudit()` 裡以 `wsKept=` 露出來對照,沒上面板。
- 程式註解留了一句舊評語「wsBoard=0 vs colonist 11」——但那個 `wsBoard` 是 board 自己的即時值(reload 會歸零),不是持久化的 `wsKept`。真正要驗的是 `wsKept` vs `victory`。

## D 的真正範圍(兩步)

1. **驗證**:打一局(中間要有幾次被強盜卡到),局末跑 `__cstAudit()`,看每個玩家的 `wsKept=` 是否等於 `victory=`(colonist 的精確值)。
2. 分兩種結果:
   - **準** → **Promote**:讓面板的 ⛔ 遊戲中直接顯示 `state.wsBlocked`(取代「實收對比」估計),局末仍用 Victory 覆蓋,WS 未就緒前仍用估計當備援。改動小、低風險。
   - **不準** → 用那局的資料**除錯幾何/強盜追蹤**(找出 `accrueBlocked` 少算在哪),修好再 promote。

## 已知的根本限制(promote 後仍存在,要誠實標明)

- **中途才開的分頁 / 旁觀者**:沒有開局以來的累計、board 也無法回推 → 早期的卡田會漏。這條無解(伺服器不給歷史地圖)。正常「從頭看整局」的情況不受影響。
- 擲骰後 ~0.6 秒內 reload(還沒存檔)→ 那一次可能漏。視窗極小。

## 驗證需要你做的(含時機)

- **何時**:打一局正常遊戲,刻意讓強盜卡你或對手的產地幾次。
- **局末**(Victory 結算畫面出現時,這樣 `victory=` 才有值):在遊戲分頁 console 打 `__cstAudit()`。
- **執行後**:把整段貼回來。我看每行的 `block: panel=… wsBoard=… wsKept=… victory=…`,比對 `wsKept` 與 `victory`,決定直接 promote 還是先除錯。

## 不在範圍

不重做整張地圖快照系統(沒必要,1.65 的「存算好的累計」已等價)。骰子/回合時間不碰。
